[8:28] O QUE É "FAZER UM BOM TRABALHO"

Há uma pergunta meta-epistemológica aqui sobre o que significa fazer um bom trabalho. Fazer um bom trabalho como engenheiro de software é difícil. Leva anos no setor para internalizar o que é escrever código de alta qualidade, manutenível e confiável, em cima do qual seus colegas conseguem construir e que acumula alavancagem na codebase.

Os modelos, durante o treinamento, viram trilhões de linhas de código que tomam todas as escolhas possíveis sobre esses requisitos não-funcionais. Então é nosso trabalho especificá-los, escrevê-los de forma que os agentes vejam: isto é o que é fazer um trabalho aceitável que vai virar patch mergeado.

Se os agentes não estão fazendo isso, é nosso trabalho refinar e restringir a saída deles para que o código seja aceitável. Você pode literalmente dizer "não produza slop, não aceite slop" e não vai ter slop na codebase. Mas isso exige absorver perdas de velocidade no curto prazo, voltar e dar dois cliques na tarefa para entender com o que os agentes estão lutando no seu ambiente. Colocar as proteções no lugar para parar de cometer esses erros, e depois afastar-se e gastar seu tempo em atividades de maior alavancagem.

[11:05] PROMPTS, REVIEWERS E AUTO-COMPACTAÇÃO

Como fazer os agentes trabalharem bem? As formas de fazer isso incluem ter agentes revisores olhando o código pela lente do que é sucesso. Temos agentes de revisão de segurança e confiabilidade rodando em todo push e CI, que olham para a documentação e o patch proposto e fazem coisas simples como perguntar: tem timeout e retry nesse trecho de código de rede? O código novo tem interface segura, impossível de usar errado?

Tenho certeza que todo mundo aqui já foi paginado por código de rede que falhou em produção causando outage que poderia ter sido remediado por um retry e um timeout. Sou culpado de colocar o retry e o timeout, mergear o bug fix e ignorar o resto. Não sou um revisor nem autor de código confiável quanto a esse requisito não-funcional. Mas dedicar tempo para escrever docs, escrever um lint próprio do meu codebase que olha cada chamada de fetch para garantir retry e timeout — isso resolve o problema durávelmente. E consigo fazer porque me apoio no axioma de que código é livre, que os agentes fazem um bom trabalho e que posso migrar a codebase inteira para resolver isso uma vez por todas.

Para operar assim, precisamos olhar para classes duráveis de falha que agentes e humanos cometem repetidamente. Entender por que estamos gastando tempo nelas. Bolar uma solução para eliminar sistematicamente essa classe de mau comportamento. E continuar observando, refinando e fazendo escolhas adicionais sobre os requisitos não-funcionais.

Sem mostrar de verdade: como é seu workflow? Como você ataca uma tarefa?
— A forma como eu e meu time trabalhamos é começar com tickets. Temos pedaços de trabalho que queremos fazer, features para adicionar nos apps, trabalho de confiabilidade. Damos esse ticket para um agente junto com um conjunto de skills que permitem manipular nosso app. Queremos que o ponto de entrada do desenvolvimento seja o Codex, não um ambiente que construímos em volta dele. Fazemos as coisas de fora para dentro: o Codex é o ponto de entrada do mesmo jeito que você seria.

Damos ferramentas e instruções de como cozinhar. Em vez de criar um shell onde nosso app e o Codex são spawnados, temos uma skill que ensina o Codex a subir o app, a subir o stack local de observabilidade para ter logs e telemetria. Damos uma skill que permite bootar o Chrome DevTools e anexar à aplicação via uma CLI local que conecta por um daemon.

Todo o jeito como organizamos o repositório e as ferramentas locais de dev é para o Codex invocá-las primeiro. Isso significa que temos uma série de mini-harnesses dentro da codebase que facilitam encaixar guard-rails adicionais. Um pacote grande de regras ESLint customizadas conectadas em todo pacote PNPM do workspace. Um harness local de dev que permite adicionar testes de mais alto nível, "wholesome", que afirmam a estrutura do próprio código — e não só sintaxe ou comportamento. Coisas como privacidade de pacote, arestas de dependência entre camadas do stack, garantir que entre vários arquivos os schemas zod sejam dedup, que exista uma única implementação canônica dos helpers async.

Por que isso? Porque a gente vê os agentes às vezes otimizarem por coerência local de um pacote em vez de usar utilitários compartilhados. Tendo observado, construímos um monte de mini-linters e verificadores de código-fonte que arrancam esse comportamento ruim, para que humanos não percam atenção em review com isso. O setup otimiza para o agente fazer o trabalho e para humanos não terem que acompanhar a alta rotatividade do código.

Centralizamos a alavancagem em umas 5 a 10 skills. Não vamos largo em skills — preferimos melhorar as existentes. A infraestrutura do repositório, todas as ferramentas locais de dev, mudam super frequentemente, e eu não tenho banda para acompanhar. Então escondemos toda essa complexidade atrás das skills que o humano invoca, e deixamos o agente descobrir.

Uma coisa legal: quando trocamos de usar Chrome DevTools Protocol direto para usar essa coisa de daemon, eu nem sabia que tinha mudado por umas três semanas. Estava tudo bem, o Codex conseguia fazer o que precisava com a documentação que tínhamos no lugar.

[23:35] COMO EVITAR OVER-ENGINEERING DE HARNESSES

Você escreveu um artigo chamado "Harness Engineering" com uma seção sobre milhares de skills versus simplificar para poucas. Como você se impede de overengineerar harnesses? E você constrói ferramentas pequenas para si mesmo?

— Acho que isso aponta na direção da bitter lesson: como garantir que o trabalho que faço não fique completamente obsoleto com aumento de capacidade do modelo? Penso fazendo o mínimo de gerenciamento de contexto necessário para puxar requisitos para o agente fazer um trabalho aceitável ao longo do percurso. Contexto é algo que não vai ficar obsoleto — os modelos precisam ser informados sobre os requisitos da tarefa, sobre quais guardrails prestar atenção.

Um bom harness é operacionalizado para entregar texto ao modelo na hora certa, para que ele olhe o trabalho que fez e a informação sobre o que é um bom trabalho. Fundamentalmente os modelos são treinados para seguir instruções. Tudo o que o harness deveria fazer é fazer aflorar instruções no momento certo. Você quer minimizar isso também — não dá para frontload tudo, porque sobrecarrega o agente. Mas todos esses requisitos precisam ser atendidos durante todo o percurso de um PR. Adiar ou trazer just-in-time essas instruções é o que um bom harness deve fazer.

Se você sabe que quer seus componentes React decompostos para gerar bons snapshot tests com peças mais sem-estado, não precisa carregar isso no início. Deixa o agente cozinhar e prototipar a UI; em tempo de lint ou teste, fala: "agora que terminou, separa para os componentes ficarem pequenos, sem estado, com dependências locais em hooks em vez de prop drilling". O agente diz "ok, instrução nova", pega o patch, adapta e sobe para o GitHub. Esse tipo de coisa não vai ficar obsoleto com aumento de capacidade. Trata-se de levar o texto certo, no contexto certo, na hora certa.

[30:06] POR ONDE COMEÇAR COM AGENTES DE CÓDIGO

Como pessoas que escrevem muito código manual deveriam transicionar? Eu ainda checo todo PR, copio e colo do Codex. Como o engenheiro médio começa a usar essas ferramentas?

— Acho que tem dois jeitos. Um é começar usando os agentes de código para melhorar sua confiança no código existente. Acho que todo mundo concorda que mais testes provavelmente é coisa boa — afirmar que os programas estão bem especificados e se comportam corretamente conforme os usuários interagem é coisa boa. Os agentes são super bons em olhar o código existente com algum contexto de como ele deve ser usado e escrever testes que afirmam esse comportamento. O outro jeito é olhar como você está gastando seu tempo. É olhando para o editor escrevendo código? Esperando testes rodarem? Esperando feedback de review humano? CI lento? Talvez você tenha um monte de testes flaky. Use os agentes para automatizar incrementalmente as partes onde você está gastando tempo. As partes de maior alavancagem do nosso trabalho são definir o que precisa ser feito, priorizar e agendar, e empoderar os colegas a fazer. Quanto mais a gente delega e vai para esse papel de sequenciamento e orquestração — pensa em tocar times — mais execuções paralelas e profundas dessas delegações conseguimos. Se eu coloco primitivas que facilitam responder a eventos numa fila Kafka, não preciso ficar no detalhe com cada engenheiro implementando consumer corretamente. Esse mesmo estilo de building block aplica super bem aos agentes e empilha super bem.

[33:39] ESCALANDO ARQUITETURA — 750 PACOTES PNPM

Conforme seu mapa de conhecimento da org escala, que passos práticos você tem para divulgação progressiva? Como escalar agentes para trabalhar bem com codebase maior?

— Quando comecei o projeto: repositório em branco, Electron app, V single package. E acabamos numa bagunça, porque não tinha privacidade de pacote para impor invariantes sobre quais APIs são públicas. O agente não tinha hooks concretos no sistema de arquivos para determinar quais domínios estavam separados. Acabamos indo de cara para "organização de 10.000 engenheiros", pesado em arquitetura: 750 pacotes no workspace PNPM, isolados por domínio de regra de negócio ou camada do stack, pacotes pequenos individuais util que encapsulam funcionalidade reusável e que linta-se sobre o uso, codificando alavancagem.

Mesmo que você não tenha microservices, estruturar repositórios para conseguir escopar o subree do diretório que está olhando para conseguir fazer a maior parte da mudança, ajuda. Código no sistema de arquivos também é texto — então é efetivamente prompt que você está dando para seu agente de código. Fazer o código ser o mais parecido possível faz com que, independente de onde no repositório seu agente esteja olhando, ele desenvolva muito contexto transferível. Você deveria ter um único jeito de fazer um helper de bounded concurrency. Um único jeito de construir um comando observável com efeito colateral instrumentado. Um único ORM. Uma única linguagem de programação. Um único jeito de escrever scripts de CI. Um único jeito de adicionar regras de lint. Isso significa que os tokens que você quer que o modelo produza são mais fáceis e mais consistentemente previstos, independente de onde ele olhe.

Estruture o código para ser local a um subree na maioria das interações com aquele sistema. E descubra como usar agentes para migrar a codebase inteira para ser igual. Empodere alguém do time para ser ditador e dizer "é assim que tem que ser feito". Escreve, evolui o código para refletir essa realidade.

[36:26] CODE REVIEW EM ALTA VELOCIDADE — GARBAGE COLLECTION DAY

Como você aborda code review agora que tem tanta velocidade? Você simplesmente não lê o código? Confia no test coverage? Como remove o bloqueio mental de "preciso checar tudo manualmente antes de mergear"?

— Mesma ideia: olhe onde está gastando tempo, descubra como gastar menos. Quando começamos, o primeiro passo foi fazer o agente produzir código aceitável de forma confiável. E o desafio grande: cada engenheiro produzindo 3 a 5 PRs por dia, mesmo num time de 3, conflitos de merge ficaram terríveis. Os PRs eram grandes, trabalhávamos nas mesmas partes. Aí movemos em duas direções: arborizar mais o código para minimizar conflitos, e minimizar o tempo que PRs ficavam abertos. Os PRs ficavam muito tempo abertos porque precisavam de code review — humanos eram o gargalo.

Para automatizar isso, pedi a cada engenheiro do time para tirar um dia por semana — sextas, chamamos de "garbage collection day" — em que o trabalho inteiro era pegar todo bit de slop observado durante a semana que estava dificultando merges e descobrir como eliminar categoricamente daquilo voltar a acontecer. Foi quando começamos a fechar o loop entre o feedback que humanos davam (que indica falha de contexto do agente), levar isso para o repositório e descobrir como prompt-injetar o agente automaticamente para que se autocurasse. Vai do tempo síncrono humano gasto dando feedback como comentário de review para documentação no repositório, surfada automaticamente via teste falhante ou agente revisor primado para revisar com base nos docs. Tudo isso acontece colocando esses docs num único lugar a que todos esses processos conseguem se acoplar.

Pedimos para as pessoas categorizarem o tipo de feedback de review como persona — tipo arquiteto front-end, engenheiro de confiabilidade, escalabilidade. Para cada persona, subimos um agente revisor disparado em cada push, que pergunta "esse código está bom? Aflorar P2 ou pior que bloqueariam o merge baseado nesta documentação que diz como é bom". Apendando continuamente nesses arquivos, vimos slop reduzir, reduzir, reduzir.

[41:49] CÓDIGO COMO ARTEFATO DESCARTÁVEL — LLM COMO COMPILADOR FUZZY

Código é um artefato de build descartável?
— Sim. Tocamos nisso com o Symfony, esse orquestrador de agentes que lançamos. A ideia de que dá para publicar uma biblioteca que é, na real, uma especificação muito bem definida, da qual o código é o artefato compilado. Usar LLM como compilador fuzzy é um modelo mental interessante. Todo o contexto que estamos pondo na codebase para harness engineering é, efetivamente, restrições e passos de otimização sobre qual código é aceitável construir em primeiro lugar. É bem similar às passadas de análise estática e otimização que algo como o LLVM faria compilando código Rust.

Trocar um modelo por outro é como trocar o backend de geração de código de LLVM para Cranelift no compilador Rust — você espera que todas as regras sobre o que é Rust aceitável produzam código de máquina válido e sólido na saída, mesmo que o processo de geração mude e você acabe com instruções x86 diferentes. Mesma mentalidade para LLMs trocando modelos. Queremos que a estrutura ao redor do código limite o jeito como é escrito ao que seria aceitável para nós.
