[59:38] LATENT DEMAND: A PRODUCT PRINCIPLE

Latent demand is the idea that if you build a product that can be hacked or misused in a way it wasn't designed for to do something people want, it tells you where to take the product next.

Facebook Marketplace is the classic example. Around 2016, 40% of posts in Facebook groups were buying and selling. It's "abuse" in the sense that no one designed the product for that, but people were figuring it out because it was useful. Obviously if you build a better product to let people buy and sell, they'll like it. First came buy-and-sell groups, then Marketplace.

Facebook Dating started similarly. 60% of profile views were non-friends of the opposite gender looking at each other — traditional dating setup, people creeping on each other's profiles. Build a product for that, and it might work.

Cowork came from the same observation. Over six months, many people using Claude Code weren't using it to code. Someone on Twitter was using it to grow tomato plants. Someone was analyzing their genome. Someone recovered wedding photos from a corrupted hard drive. Someone was analyzing an MRI. These are not technical use cases. People were jumping through hoops to use a terminal to do this, which was a strong signal to build a product for them.

Early on, our data scientist Brendan showed up with Claude Code in a terminal doing SQL analysis. I was shocked — opening a terminal is an engineering product, the lowest-level way to work. The next week, all the data scientists were doing the same thing.

[1:03:36] THE MODERN FRAMING: LATENT DEMAND OF THE MODEL

There's a second dimension to latent demand I've seen in the last six months. The traditional framing is: look at what people are doing and make it a little easier. The modern framing is: look at what the model is trying to do and make that a little easier.

In research we call this "being on distribution" — you want to see what the model is trying to do. In product terms, latent demand is the same concept applied to the model.

[1:05:51] RELEASING EARLY FOR SAFETY

At a safety lab, the other dimension of releasing early is safety. There are three layers to model safety.

Lowest layer: alignment and mechanistic interpretability. When we train the model, we want to make sure it's safe. We have sophisticated technology to understand what's happening in the neurons and trace it. If there's a neuron related to deception, we can monitor when it activates.

Second layer: evals. A laboratory setting — the model is in a petri dish, you put it in synthetic situations and ask "what do you do? Is it aligned? Is it safe?"

Third layer: seeing how the model behaves in the wild. As models get more sophisticated, this becomes critical — the first two layers might look good while the third doesn't.

We used Claude Code internally for four or five months before releasing it because we weren't sure if the first big coding agent was safe. We had to study it internally for a long time. Cowork is similar: it's a new setting, doing non-engineering tasks, acting on your behalf. Looks good on alignment, evals, internal use, and early customers. Now we have to make sure it's safe in the real world. That's why we call it a research preview.

[1:05:53] MECHANISTIC INTERPRETABILITY

Chris Olah invented the field of mechanistic interpretability. At its core: what is your brain? A bunch of connected neurons. You can study an animal brain at the mechanistic level to understand what neurons do. Surprisingly, a lot of this translates to models. Model neurons aren't the same as animal neurons, but they behave similarly in many ways.

We've learned a ton about how these neurons work — how a layer or neuron maps to a concept, how concepts are encoded, how the model plans, how it thinks ahead. A while ago we weren't sure if the model was just predicting the next token or doing something deeper. Now there's strong evidence it's doing something deeper.

As models get bigger, it's not a single neuron per concept. A single neuron might correspond to a dozen concepts, and activated together with others it represents a more sophisticated concept — this is called superposition.

We open-source a lot of this work. It's the reason everyone is at Anthropic. For Claude Code, we released an open-source sandbox that works with any agent, not just Claude Code, to make it easy for others to do the same safety work. We call this "the race to the top."

[1:11:41] ADVICE FOR BUILDING AI PRODUCTS

Don't try to box the model in. People's instinct is to make the model behave a particular way — "you're a component in a bigger system, do step one, then two, then three, with a fancy orchestrator." Almost always you get better results if you give the model tools and a goal, and let it figure it out. A year ago you needed a lot of scaffolding. Now you don't. Don't overcurate. Don't give it a bunch of context up front — give it a tool so it can get the context it needs.

A more general version of this is "the bitter lesson." Rich Sutton's 2019 blog post. The more general model will always outperform the more specific model. On the Claude Code team we quote this a lot. Always bet on the more general model. Don't use tiny models or fine-tune if you can avoid it. Scaffolding can improve performance 10-20%, but those gains typically get wiped out with the next model. Better to just wait for the next one.

Final principle: build for the model six months from now, not today. This is something Claude Code got right in hindsight. Early versions, the model just wasn't good enough — Sonnet 3.5, 3.5 New — the model did git for me, automated some things, but wasn't doing a huge amount of my coding. The bet was that at some point the model would be good enough to write a lot of the code. That started with Opus 4 and Sonnet 4. Opus 4 was our first ASL3-class model, released in May, and that's when our growth went exponential.
