LOCAL OU NUVEM? O LLM CODING BENCHMARK 6:11

Primeira pergunta clássica: rodar LLM local ou na nuvem? Você vai ver 500 milhões de opiniões por aí. Resolvi fazer um teste de verdade. Tem um projeto no meu GitHub chamado LLM Coding Benchmark.

Eu não gosto dos benchmarks oficiais. A maioria são pequenos trechos de código muito simples, o que a gente chama de LeetCode: “como ordena uma lista”, “como muda essa cor para aquela”. Não é projeto. Então automatizei o Open Code, que é a versão open source de Codex e Cloud Code, e rodei bateria de testes em mais de 33 modelos. Open source local na minha máquina, e as versões comerciais desses mesmos modelos na nuvem via OpenRouter. O OpenRouter é uma API única onde você paga uma vez e acessa Claude, GLM, Minimax, Kimi, DeepSeek, o que quiser.

Pedi para cada modelo fazer um mini chat web tipo ChatGPT que se conecta ao OpenRouter. É nível Hello World, entrevista de emprego de programador júnior, um ou dois dias no máximo. Dois prompts: um para gerar a primeira versão, outro para fechar o projeto. Dá para rodar em todos.

O HARDWARE: RTX 5090 E MINI PC AMD RYZEN AI MAX 9:00

Para rodar tudo localmente, eu uso uma RTX 5090 da Nvidia, a melhor placa do mercado. O problema: 32 GB de VRAM só. Não rodam modelos acima de 30 bilhões de parâmetros. Para os maiores comprei um mini PC com AMD Ryzen AI Max, 128 GB de RAM, dos quais 96 GB posso dedicar à GPU integrada. Dá para rodar Qwen 3 de 122 bilhões de parâmetros.

Se você tenta rodar um modelo grande numa máquina que não aguenta, ele vira carroça. A GPU consome toda a VRAM, começa a usar a RAM da CPU, fica transferindo camadas entre RAM e VRAM usando a CPU no meio do caminho. Lento. É um swap, igual swap de disco quando você não tem RAM. Sem mágica.

POR QUE OPUS É MELHOR: A CURVA EM S 14:30

O que faz Opus ser Opus? Durante 2022-2024 foi a corrida de parâmetros, a guerra dos parâmetros. Chegamos a 1 trilhão. Só que 10 vezes mais parâmetro não vira 10 vezes mais inteligência. É a tal da curva em S. A partir de 2024, o ganho marginal virou inviável: gastar 10 vezes mais energia, 10 vezes mais máquina, para ganhar duas vezes de qualidade. Para ir de 1 trilhão a 2 trilhões, nem energia no planeta tem.

Então o jogo mudou. Pararam de subir parâmetro e começaram a otimizar as técnicas por cima do mesmo modelo.

O NASCIMENTO DO THINKING 16:32

Em 2024 a OpenAI lançou o O1, o modelo omni. A grande sacada dele foi o thinking. Uma LLM gera texto calculando a tensão em cima do seu contexto para gerar o próximo token, e repetindo a conta. Antes do thinking, ele gerava resposta direto, e as respostas eram burras.

Com thinking, ele é treinado para, antes de responder, gerar novas perguntas para si mesmo. Fica se retroalimentando. Aquele momento em que o chat mostra “Thinking...” é ele gerando perguntas e devolvendo para ele mesmo, camada após camada de resposta. Deep thinking ou high thinking é quantas camadas profundas ele pode ir. Por isso Opus é mais lento que Sonnet: pensa mais fundo. E por isso gasta mais token.

MINI, HAIKU, SONNET, OPUS 17:35

Os modelos mini ou rápidos têm pouco ou nenhum thinking. Os pesados têm muito. Na OpenAI, Mini é o rápido. Na Anthropic, Haiku. No meio, GPT normal e Sonnet. No topo, GPT 5.4 High, Opus, e agora o Mitos. Quanto mais deep thinking, mais token, mais custo, mais tempo de espera. É você que decide.

Todo bom programador tem que ser promíscuo e desleal com toda tecnologia. Nunca fui outdoor de linguagem de programação. JavaScript, Python, Java — o que resolve o meu problema melhor naquele momento, é o que eu uso. LLM é a mesma coisa.

O PROBLEMA NÃO É O MODELO, É O HARNESS 22:38

Hoje conversei com o Mr. Kibs, um conhecido que foi para a OpenAI. Ele tentava me evangelizar no Codex. Eu disse: cara, por que o Codex não atende 100% do que o Cloud me faz? O problema não está no modelo, está no harness. Harness em português é rédea: a corda que se põe no cavalo para controlar. O modelo é um componente, o harness é a ferramenta que usa o modelo. Os dois trabalham juntos.

O Cloud Code neste momento tem melhor harness do que o Codex. Se o Codex implementar a mesma estratégia usando o mesmo GPT, provavelmente vai ter respostas iguais ou melhores. Por isso eu não me dou ao trabalho de escrever livro sobre isso: hoje tem uma versão, amanhã pode sair o Codex novo e mudar o jogo.

O CÓDIGO VAZADO DO CLOUD CODE 23:50

O Codex da OpenAI é aberto, está no GitHub. O Cloud Code sempre foi fechado. Semana passada teve o vazamento do Cloud Code. Todo mundo foi ver como foi feito. Eu fiz uma cobertura gigante disso no blog, explicando passo a passo. O código é macarrônico, mal feito. Empurrando código o mais rápido possível para sair na frente. Mesmo assim, são 500 mil linhas de código. É um projeto médio. Cloud Code é Cloud Code feito por Cloud Code: os programadores da Anthropic fazendo a ferramenta nela mesma.

A BRIGA ANTROPIC VS OPENAI 25:42

A briga é como Netflix versus Amazon Video versus Disney+. Eles querem absorver o máximo de gente agora como assinante. Quem assina Claude não vai querer pagar também OpenAI. Então cada lado está brigando pelo que a gente chama de market share. Não podem se olhar na mesma rua. Isso é ótimo, porque é o que força inovação acelerada.

CLOUD É MAIS DISCIPLINADO QUE CODEX 26:40

Nos projetos que testei, Cloud tem sido mais disciplinado, mais organizado, mais previsível que Codex. Codex tende a não terminar tudo que pedi, especialmente se for coisa mais complexa. Se peço coisas em paralelo, às vezes ele não faz tudo. Se interrompo, ele não volta para a primeira. Tenho que lembrar.

Cloud consegue quebrar tarefa complicada em plano, faz em tarefas paralelas. Se interrompo com algo novo, ou ajusta ou cria tarefa paralela e continua tudo em paralelo. No fim, reporta “terminei tudo”. Confio em deixá-lo rodando em um enquanto vejo outro. Isso é a versão de abril de 2026. Semana que vem pode inverter.

O NOVO VOCABULÁRIO 28:51

Para acompanhar, precisa aprender vocabulário: IA é o guarda-chuvão. Dentro, LLM, computer vision etc. Dentro de LLM, harness é a rédea, a ferramenta de linha de comando. Dentro do harness, existem skills. Skill é uma habilidade: você pede ao Claude para salvar um procedimento como skill, e a partir daí ele carrega aquele contexto toda vez que for relevante. É “rédia dentro da rédia”.

PROMPT, SYSTEM PROMPT E TOOL CALLING 30:53

LLM é um arquivão que sobe na memória da GPU para calcular o próximo token. Para manipular o modelo, você manda um prompt. Tem vários tipos. O que você não vê se chama system prompt. A rédea abre a sessão já com um system prompt: “você é um assistente de programação, tem acesso às ferramentas A, B, C, D; toda vez que o usuário fala X, você responde Y”. Já tem série de instruções pré-carregadas.

Daí você manda seu prompt. Vai via API para o LLM, que faz deep thinking. Ele devolve um plano. Você aprova. Ele começa a executar. Nesse ponto entra o tool calling: treinamento que começou a ficar padrão em 2024-2025. Quando o modelo precisa criar um arquivo, ele devolve uma estrutura com a função `createFile(“index.html”)`. O harness recebe essa estrutura e executa na sua máquina. O LLM não executa nada. Quem executa é o harness.

COMO O HARNESS EXECUTA 32:55

Se o LLM quer ler um diretório, ele manda `listDir(“/”)`. O harness executa, captura a saída e devolve como novo prompt para o LLM. Antes de 2024, a gente fazia isso manualmente: copiava o erro, colava no chat, esperava resposta, copiava o código, colava no terminal. Tool calling automatizou tudo. O harness é burro: só recebe função, executa, devolve. A inteligência toda está no modelo, mas a execução fica no harness. Por isso harness precisa ser bom.

CADA HARNESS FALA SUA LÍNGUA 35:41

Cada empresa define o formato proprietário de tool calling. Na OpenAI chama `create_file`, na Anthropic `file_new`, na GLM `new_file`. É como se fosse Java, Python e JavaScript. O harness tem que saber falar tudo. O Open Code, que é open source, tenta falar todos ou pede ao modelo para devolver no formato que ele entende. Se o modelo não foi treinado naquele formato, ele tenta traduzir, a performance degrada.

Por isso Cloud Code com Opus, Codex com GPT, Gemini com Gemini CLI ou Antigravity. Tecnicamente dá para misturar, mas estaria sendo burro.

A ANTROPIC BLOQUEIA HARNESS DE TERCEIROS 38:26

A Anthropic tem planos de assinatura e tokens avulsos. Os planos Pro, Max, Max 20X permitem Cloud Code. Mas você não pode usar o plano com Open Code, o open source. Mesmo se Open Code fizer engenharia reversa e suportar a API, a Anthropic bane. É como trapaça num jogo com anticheat.

O token avulso é o dobro do token do plano Max. Para cada dólar que você paga no plano, paga dois se for avulso. Na OpenAI é pior: o equivalente ao Max 200 em token avulso daria 900 dólares antes da mudança recente. Seis, sete vezes mais. Então plano é muito mais barato, e a Anthropic é agressiva nas restrições justamente para empurrar assinaturas. Além disso, o plano guarda o contexto do projeto, o que é valioso por si só.

SESSION IDS: CONTINUE NO CODEX O QUE COMEÇOU NO CLOUD 40:51

Quando você dá Ctrl+C no Cloud Code ou Codex, ele mostra um session ID. Dá para voltar à mesma sessão com `cloud resume `. E o Codex aceita continuar uma sessão do Cloud: basta apontar o ID. Os dois gravam o contexto em arquivo texto. Pulo entre os dois em projetos ativos: “Cloud travou, pega de onde parou e tenta”.

AI JAIL: ROOT, YOLO E MÁQUINA VIRTUAL 42:22

Esses harness têm acesso a tudo. Podem, em tese, apagar seus arquivos. Em centenas de horas nunca vi acontecer. Em geral eles trabalham na pasta que você apontou e pedem aprovação para tudo. Se você quer pular a aprovação, tem a opção `--yolo` (you only live once). Uso yolo na minha máquina pessoal. Nunca em máquina de trabalho ou de cliente.

Um dos meus projetos abertos chama AI Jail. Usei ele quando fui configurar um servidor novo do zero. Instalei Linux, instalei Cloud, dei root, e fui pedindo por prompt para configurar a máquina. Ele virou minha ferramenta de provisionamento.

AS RESTRIÇÕES E O OPEN CLAW COMPRADO PELA OPENAI 43:43

Recentemente, a Anthropic cortou o Open Code. Muita gente usava Open Code com cloud. Em seguida apareceu o OpenClaw, outra engenharia reversa. Todo mundo foi instalar OpenClaw. A Anthropic cortou também. E aí a OpenAI comprou o OpenClaw. O cara que criou o Open Claw hoje trabalha na OpenAI. A Anthropic: “não vou deixar rodar o produto do meu concorrente no meu plano”. Direito deles.

A GUERRA DE PARÂMETROS ACABOU 46:25

Não dá mais para ir de 1 trilhão a 2 trilhões. Precisaria de 20 usinas nucleares a mais. Melhor gastar esse tempo ajustando o modelo que existe. Por isso a OpenAI foi iterando em 5.0, 5.1, 5.2, 5.4. É ajuste, não novo modelo. Antes 70% do compute de data center era treinamento do próximo frontier model. Essa matemática inverteu: hoje 70% é inferência (o uso no chat) e falta data center.

Acompanhar não é microgerenciar. Não fica no cangote. Mas passa, olha, pergunta. Vai ouvir “ah, eu ia fazer, era a próxima coisa”. Ia mesmo ou ia tampar?

A IA É NEUTRA 4:07:28

A parte boa: a IA é neutra. Não tenta te agradar por carência, não tem malícia, não quer te enganar. Tenta executar o que você pediu com os dados disponíveis. Não fica com raiva se você gritar, não cospe no prato se você xingar. Libertador. Pode testar o limite sem drama. Se mesmo assim a resposta está errada, é porque você não soube pedir.

METODOLOGIA PARA COMPARAR MODELOS 4:08:25

Comparei os modelos open source de tradução em mesmos termos. Mesmo hardware, mesmos prompts, mesmos resultados checados, mesmas dimensões: tokens consumidos, tempo levado, qualidade do output. Ao final, não concluí “este é melhor”. Concluí: “este é melhor se você tem dinheiro”; “este outro é melhor, mas dá mais trabalho”; “este é melhor, mas leva o triplo do tempo”.

Analogia do carro. Não tem carro que resolve todo problema. Para entrega de tofu na montanha, meu AE86 serve. Para velocidade e dinheiro, pego Ferrari. LLM é igual. Não existe LLM que resolve tudo. Não existe linguagem perfeita, framework perfeito. Seja promíscuo e desleal com toda tecnologia.

NÃO PRECISA DAR BOM DIA 4:09:51

Não precisa dar bom dia para a IA. Gasta token. Tem o meme do apocalipse em que a IA perdoa quem foi gentil. Para cada bom dia que alguém dá, eu faço bullying 30 vezes. Nunca vai funcionar. Seja direto. Use todas as tecnologias disponíveis.

ROBSON: LLM VAI SER ENGENHEIRO DE SOFTWARE? NÃO 4:10:11

Não. Vai ser ferramenta do engenheiro. Sem engenheiro de verdade, o código explode em produção. “Mas eu crio uma persona 'você é engenheiro de software'”. Zero garantia. Não tem contexto, não vê clientes, não ouve reclamação, não sabe urgências e prioridades. Se você pede “seja o engenheiro mais perfeito do mundo”, ele faz código hiperburocrático, pesado pra caramba, para atender dez pessoas. Se pede “faça o mais seguro possível”, bloqueia tanto que não instala. Engenheiro de software estuda anos, apanha para saber que métricas importam. Reconhece: “já vi isso dar problema, não vai dar agora mas vai dar amanhã, deixa pré-ajustar”. Esse tipo de experiência nasce de apanhar. E aprendizado tem que ser deliberado, como personal training: pacing, estratégia, resistência. Não é só correr no talo.

CONTADOR, PREENCHEDOR DE PLANILHA E EXCEL 4:11:45

Antigamente, empresa de contabilidade ocupava andares inteiros com gente preenchendo pilhas de planilhas de papel. “O Excel matou o contador.” Não matou. Contador existe até hoje. O Excel acabou a função.

Mesma coisa agora com programação. Acaba o codificador: o cara que passava o dia trocando cor de botão, cortando imagem no Photoshop, recortando HTML à mão. Trabalho braçal, nunca foi programação, já devia estar automatizado. O engenheiro que analisa, decide, se responsabiliza, continua. O problema é que muito programador ruim botava “engenheiro de software” no LinkedIn. Você nunca foi engenheiro de software. Esse é o emprego que a IA rouba.

MICHAEL: CLOUDFLARE, NEXT.JS E WORDPRESS CLONE 4:16:23

Michael pediu para eu falar sobre a Cloudflare reimplementando a API do Next.js. A Vercel dava suporte privilegiado ao Next.js: o framework é bem incompleto sem o harness/rédeas que a Vercel acoplava. Era lock-in de anos. Dava para hospedar em outro lugar, na Vercel era trivial.

A Cloudflare, concorrente, reimplementou essas rédeas usando IA. Em uma semana ficou pronto. Lançaram o produto e o tutorial de como migrar da Vercel para a Cloudflare.

Mesmo tipo com WordPress. Um dos piores códigos que existe. Barata. Parece que nunca morre. Código ruim, inseguro, cheio de plugins mal feitos que abrem buracos. A Cloudflare usa vários produtos de segurança só para interceptar vulnerabilidades do WordPress antes que cheguem no servidor. Todo mundo já tentou clonar WordPress; todo mundo falhou. A Cloudflare fez clone (chamado MD-EM dash) em linguagem melhor, com IA, e está empurrando como substituto.

O DEVELOPER 10X AGORA É REAL 4:19:04

Todos os projetos que mostrei foram feitos em 2-3 dias. Importante calibrar: são 2-3 dias meus, em 16 horas concentradas, não 8. Ninguém programa 8 horas. A gente trabalha 8, mas metade é e-mail, reunião, Slack, procrastinação. Num bom dia, 4 horas de código. Normal é 2 ou 3.

No ritmo normal, pelo menos 4 vezes mais tempo para sênior. Nível pleno, mais 4 vezes. Minha versão em 4 dias, outra pessoa em 2-3 meses. Dois meses para básico, mais um mês para testes, escalabilidade, documentação. Antes era difícil pegar projeto open source e encaixar na empresa. Alguém tinha que dedicar tempo para ajustar. Hoje eu pego, IA ajusta rápido. Em vez de terceirizar para 10 pessoas, pego um sênior in-house, dou uma semana. Vale por 10. O “developer 10x” virou realidade. Agora. Em novembro ainda não era. Por isso a urgência.

BLOG DE 20 ANOS TRADUZIDO EM DOIS DIAS 4:22:03

Em 5 de abril meu blog fez 20 anos. Mais de 700 posts, quase 1000 contando os que tirei por estarem obsoletos. Majoritariamente em português. Amigos fora do Brasil sempre diziam “queria mostrar para meu time”. Google Translate é bosta. Agora, com o estágio atual das LLMs, finalmente tradução não parece máquina. (A propósito, estudos de LLM nasceram de tradução: large language models. Se traduz alemão bem, sabe programar.)

Dois dias. Ajustei o template com links em inglês e português. Post que era inglês, foi para a seção em inglês. Os em inglês traduzi para português. O resto traduzi para inglês. Traduzindo com o Claude. Gastei metade do meu Max 20x e traduziu o blog inteiro. Lendo em inglês, parece texto normal, não IA.

Conheci pessoalmente Chris Wanstrath fundador do GitHub antes dele ser GitHub, quando ele era programador de Starbucks. David Heinemeier Hansson, criador do Ruby on Rails. Trouxe todos para o meu evento em 2008, ano do primeiro. GitHub só virou GitHub em 2010. Microsoft comprou 5 anos depois. Tem muita história boa no blog.

146 VÍDEOS DUBLADOS EM QUATRO DIAS 4:25:03

Cinco anos de vídeos, 146 vídeos, quase 100 horas. Se dublasse, tinha que bater o timing do áudio original para os elementos visuais não saírem de posição. Se regravasse, precisava manter duração. Parecia inviável.

Testei modelos de TTS. Open source não tinha entonação que eu queria para inglês. Fui para comerciais. Vídeo de uma hora você não envia inteiro, tem que picotar em trechos de menos de 700 palavras, gerar, reconectar, pegar o timing.

Peguei timing do YouTube, peguei meus scripts, recortei. Legenda mostra duas linhas por vez. Sem os scripts, não saberia onde termina parágrafo, colocaria silêncio no meio de frases. Em vídeos com Steve Jobs (trecho embutido), YouTube alucinou a legenda. Comparei script original com legenda, nos pontos que não batiam peguei o áudio original.

Título, descrição, legendas, áudio com minha voz clonada na Eleven Labs, thumbnails com Nano Banana. Custo total (Claude Max + Eleven Labs): uns dois mil dólares. Entonação da minha voz clonada ainda não é perfeita, pareço um pouco de máquina porque não forcei as tags de emoção. Tem tag de surpresa, desprezo, alegria, risada. Dá para pôr.

O 42 DO MOCHILEIRO E A CODE MINER 42 4:33:30

Douglas Adams previu tudo. No Guia do Mochileiro das Galáxias, uma grande IA é construída, a maior do universo. Os povos primitivos perguntam: “qual a resposta para tudo?”. “Como assim, 42? Esperamos um milhão de anos para 42?” A máquina: “A pergunta que você fez me levou a 42. Se queria outra resposta, devia ter feito outra pergunta”. É tudo que eu venho falando. IA responde o que achou adequado para a pergunta. Sua pergunta é uma bosta. Essa é a moral. Por isso minha empresa se chama Code Miner 42. Marvin é o mascote. A ideia não é te dar a resposta. É te ensinar a fazer a pergunta.

Recebo requerimento e não sou quem faz qualquer coisa por dinheiro. Vários clientes chegaram com briefing, e eu disse: “honestamente, não vou fazer. O que está escrito aqui não faz sentido. Você vai gastar dez vezes mais do que imagina. Não vai funcionar. Volte e refaça a lição de casa. Quem montou para você não sabe o que está fazendo”. O cara fica puto. Alguns voltaram depois: “você tinha razão, repensei, agora sim”.

Dois tipos de programador ruim. O entusiasmado, que acha que vai substituir tudo por IA. E o resistente, que acha que IA não serve porque seu código é melhor. Filho, não é. Você sabe que não é. Já fez muita merda. Seu código não é tão bom. Eu sei. Você sabe. Não vamos fingir.
