Turning hours-long podcasts into articles you can actually read

Rodolfo Moraes · 2026-04-14

Disclaimer: I built this project to test development concepts and techniques with Claude Code and to put into practice what I've been learning. It's a personal lab that turned out to be useful for me, and I document it here for anyone interested in how a pipeline like this gets assembled in practice.

Long video is a hostile format for almost anything other than sitting and watching start to finish. A 2-, 3-, 4-hour podcast on YouTube has a low chance of being consumed afterward by the very audience that was interested.

Video is bad for:

  • Searching: no Ctrl+F. Finding "the bit where X said Y" means scrubbing through timestamps manually.
  • Quoting: timestamp links are a poor workaround, and transcribing by hand is tedious.
  • Re-reading: no visual markers, no progress, no named sections.
  • Skimming: you can't glance at it and decide if the time investment is worth it.

Articles solve all of this. But almost no podcast has a decent written version, and YouTube's auto-captions are unreadable — walls of text with no punctuation, full of "uh, like, you know, so".

The project's goal

adhenawer.net is a static site that takes YouTube or Twitter/X videos and turns them into bilingual articles organized by thematic sections. Brazilian Portuguese and English, same content, canonical URLs in each language.

Each article has:

  • Clean title and subtitle
  • Clickable table of contents
  • Thematic sections (not chronological — reorganized by topic)
  • Reading progress bar
  • "Resume where you left off" saved per device
  • References sidebar: books linked to Amazon, tools to official sites, people to Twitter, concepts to Wikipedia, related posts
  • PT/EN language switcher in the header

Zero framework. Static HTML served from GitHub Pages. Loads fast on anything, including a rough 3G connection.

The pipeline

Video URL (YouTube or Twitter/X)
  ↓
Detects provider from URL
  ↓
YouTube: youtube-transcript-api (official captions when available)
Twitter/X: yt-dlp downloads audio → mlx-whisper transcribes locally
  ↓
Raw transcript in English (or source language)
  ↓
Claude (structure): reads transcript, removes "um", "like", timestamps,
                    sponsor reads, reorganizes into thematic sections,
                    produces natural text in PT-BR and EN
  ↓
Python generates HTML with JSON-LD, Open Graph, hreflang, references sidebar
  ↓
Git commit + push → GitHub Pages deploys

No CMS, no database, no frontend build step. src/build_html.py is ~400 lines of pure Python and writes semantic HTML directly:

return f'''<!DOCTYPE html>
<html lang="{html_lang}">
<head>
<link rel="alternate" hreflang="pt-BR" href="{pt_href}">
<link rel="alternate" hreflang="en" href="{en_href}">
<script type="application/ld+json">{jsonld}</script>
<link rel="stylesheet" href="../../css/style.css">
</head>
<body class="page-article">
<header class="site-header">...</header>
<article class="container">
  <h1>{title}</h1>
  <nav><h3>Contents</h3><ol>{toc_html}</ol></nav>
  {body_html}
</article>
{references_sidebar}
</body>
</html>'''

Local Whisper

For Twitter/X videos (or YouTube when official captions are blocked) the pipeline uses mlx-whisper — the MLX implementation of Whisper, natively optimized for Apple Silicon. Transcription cost: zero. Speed: ~5× real-time on an M1 Pro. A one-hour podcast transcribes in about 12 minutes.

import mlx_whisper

result = mlx_whisper.transcribe(
    audio_path,
    path_or_hf_repo="mlx-community/whisper-large-v3-turbo",
    word_timestamps=False,
)

On Linux or Windows you can swap for faster-whisper (CTranslate2), same idea. What matters is not sending audio to anyone's API — Whisper running locally is good enough and free.

Markdown for agents

Cloudflare has a feature called Markdown for Agents that converts HTML to Markdown at the edge when an AI agent requests with Accept: text/markdown. That feature only exists on paid plans (Pro or Business).

It can be replicated in ~150 lines of JavaScript on the Free Cloudflare Workers plan (100k requests/day). Well-structured HTMLs don't need anything fancy — regex handles it:

export default {
  async fetch(request) {
    const accept = request.headers.get("Accept") || "";
    if (!accept.includes("text/markdown")) {
      return fetch(request); // pass HTML straight to humans
    }

    // Agent asked for markdown. Fetch HTML from origin and convert.
    const response = await fetch(new Request(request.url, {
      headers: { Accept: "text/html" }
    }));
    const html = await response.text();

    const articleHtml = html.match(/<article[^>]*>([\s\S]*?)<\/article>/i)?.[1];
    const markdown = htmlToMarkdown(articleHtml);

    return new Response(frontmatter + markdown, {
      headers: {
        "Content-Type": "text/markdown; charset=utf-8",
        "x-markdown-tokens": String(Math.ceil(markdown.length / 4)),
        "Vary": "Accept",
      }
    });
  }
};

Result: an article HTML that costs ~53,000 tokens for an agent to parse becomes ~13,000 tokens in Markdown — 75% reduction. No Turndown, no JSDOM (neither runs in the Worker runtime), just regex plus some care with <pre> and <code>.

You can test it:

# Humans get HTML
curl -sI https://adhenawer.net/posts/original/state-ai-2026.html | grep -i content-type
# content-type: text/html; charset=utf-8

# Agents get Markdown
curl -s https://adhenawer.net/posts/original/state-ai-2026.html \
  -H "Accept: text/markdown" | head -10
# ---
# title: "The State of AI in 2026"
# lang: en
# ---
# ## THE NOVEMBER INFLECTION POINT

References extracted automatically

Each article has a fixed sidebar with references extracted from the transcript. Categories:

  • Books → Amazon search
  • Tools → official site
  • Papers → arXiv or author's blog
  • People → Twitter/X (priority), fallback to Wikipedia
  • Concepts → Wikipedia
  • Companies → official site
  • Related posts → internal cross-link to the same theme in other articles

Extraction is done by Claude subagents in parallel — one per article, each receives the transcript and returns a structured JSON:

{
  "video_id": "...",
  "books": [
    {
      "title": "Functional Programming in Scala",
      "author": "Paul Chiusano & Rúnar Bjarnason",
      "url": "https://www.amazon.com/s?k=functional+programming+in+scala+chiusano",
      "context": "..."
    }
  ],
  "people": [
    {
      "name": "Boris Cherny",
      "role": {"pt": "Chefe do Claude Code, Anthropic", "en": "Head of Claude Code, Anthropic"},
      "url": "https://twitter.com/bcherny"
    }
  ],
  "related_posts": [
    {
      "slug_pt": "...",
      "slug_en": "...",
      "reason": {"pt": "...", "en": "..."}
    }
  ]
}

Language-sensitive fields (like role for people and reason for related posts) are stored bilingually — the renderer picks the article's language at build time.

Bilingual

Two reasons:

  1. Reach: most technical podcasts are in English, but a Brazilian audience has the language barrier.
  2. SEO: properly configured hreflang alternates in sitemap + each HTML produce natural ranking in both languages.

Canonical URLs differ on each side:

  • PT: /posts/pt_br/chefe-do-claude-code-o-que-acontece-depois-que-a-programacao-for-resolvida.html
  • EN: /posts/original/head-claude-code-happens-after-coding-solved.html

Each slug is derived from the title in its language. The header's language switcher uses <link rel="alternate" hreflang> as the source of truth — zero hardcoded "URL A → URL B" mappings.

Flow inside Claude Code

The project runs inside Claude Code (Anthropic's official CLI). In the terminal, from the repo:

> add article from https://youtu.be/VIDEO_ID

The agent executes the pipeline on its own: fetches the transcript, organizes into sections, translates to PT-BR, generates the HTML, updates docs/index.html, updates sitemap.xml, extracts references, commits, and pushes. Reviewing the diff is the human's job.

This works because of two things:

  • CLAUDE.md: a markdown file at the root containing the project's knowledge base (conventions, folder structure, slug format). It acts as persistent memory between sessions.
  • Subagents for parallelism: tasks that can be broken into independent units (like extracting references from N articles) run as parallel subagents with isolated context. Main context stays clean.

What's open

All the code is at github.com/adhenawer/video-to-text, MIT license:

  • src/pipeline.py — end-to-end orchestrator
  • src/providers/ — multi-provider abstraction (YouTube, Twitter/X). Adding a new provider (Vimeo, TikTok) requires implementing detect(url), extract_id(url), and fetch_transcript(url).
  • src/build_html.py — HTML generation
  • workers/markdown-agent/src/index.js — Cloudflare Worker
  • transcripts/ — original + translated transcripts + references JSONs
