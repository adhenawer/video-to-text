Turning hours-long podcasts into articles you can actually read

Rodolfo Moraes · 2026-04-14

I don't know if you have the same reaction, but when I see a new podcast on YouTube that's 2, 3, 4 hours long, I already know: I'm not going to watch it. Not on the phone in the subway, not on the couch after work, not during lunch. The problem isn't lack of interest — it's that long video is a hostile format for almost anything other than sitting and watching start to finish.

Video is bad for:

  • Searching: no Ctrl+F. If you vaguely remember "that guy said something about vibe coding", good luck finding it.
  • Quoting: sending a timestamp link is a poor workaround, and transcribing by hand is tedious.
  • Re-reading: no visual markers, no progress, no named sections.
  • Skimming: you can't glance at it to decide if it's worth 3 hours of your life.

Articles solve all this. But almost no podcast has a decent written version. And YouTube auto-captions are unreadable — a wall of text with no punctuation, full of "uh, like, you know, so".

What I built

adhenawer.net is a static site that takes YouTube or Twitter/X videos and turns them into bilingual articles organized by thematic sections. Brazilian Portuguese and English, same content, canonical URLs in each language.

Right now there are 20 articles published, covering people like Boris Cherny (Claude Code), Simon Willison, Steve Yegge, Andrew Ng at Stanford, Dario Amodei, Robert Shiller (Yale, on behavioral finance), Patrick Winston (MIT, that classic How to Speak lecture), and more.

Each article has:

  • Clean title and subtitle
  • Clickable table of contents
  • Thematic sections (not chronological — reorganized by topic)
  • Reading progress bar
  • "Resume where you left off" saved per device
  • References sidebar: books linked to Amazon, tools to official sites, people to Twitter, concepts to Wikipedia, related posts
  • Language switcher PT/EN in the header

Zero framework. Static HTML served from GitHub Pages. Loads fast on anything, including a rough 3G connection.

The pipeline

Video URL (YouTube or Twitter/X)
  ↓
Detects provider from URL
  ↓
YouTube: youtube-transcript-api (official captions when available)
Twitter/X: yt-dlp downloads audio → mlx-whisper transcribes locally
  ↓
Raw transcript in English (or source language)
  ↓
Claude (structure): reads transcript, removes "um", "like", timestamps,
                    sponsor reads, reorganizes into thematic sections,
                    produces natural text in PT-BR and EN
  ↓
Python generates HTML with JSON-LD, Open Graph, hreflang, references sidebar
  ↓
Git commit + push → GitHub Pages deploys

No CMS, no database, no frontend build step. The src/build_html.py script is ~400 lines of pure Python and writes semantic HTML directly:

return f'''<!DOCTYPE html>
<html lang="{html_lang}">
<head>
<link rel="alternate" hreflang="pt-BR" href="{pt_href}">
<link rel="alternate" hreflang="en" href="{en_href}">
<script type="application/ld+json">{jsonld}</script>
<link rel="stylesheet" href="../../css/style.css">
</head>
<body class="page-article">
<header class="site-header">...</header>
<article class="container">
  <h1>{title}</h1>
  <nav><h3>Contents</h3><ol>{toc_html}</ol></nav>
  {body_html}
</article>
{references_sidebar}
</body>
</html>'''

Local Whisper

For Twitter/X videos (or YouTube when the official captions are blocked) the pipeline uses mlx-whisper — the MLX implementation of Whisper, natively optimized for Apple Silicon. Transcription cost: zero. Speed: ~5× real-time on an M1 Pro. A one-hour podcast transcribes in about 12 minutes.

import mlx_whisper

result = mlx_whisper.transcribe(
    audio_path,
    path_or_hf_repo="mlx-community/whisper-large-v3-turbo",
    word_timestamps=False,
)

If you're on Linux or Windows you can swap for faster-whisper (CTranslate2), same idea. The only thing that matters is not sending audio to anyone's API to save money — Whisper running locally is good enough and free.

The part I most enjoyed building: Markdown for agents

This is the coolest one. Cloudflare has a feature called Markdown for Agents that converts HTML to Markdown at the edge when an AI agent requests with Accept: text/markdown. Problem is, that feature only exists on paid plans (Pro or Business).

I looked at their Worker runtime and realized you can replicate it in ~150 lines of JavaScript running on the Free plan (100k requests/day). The key is that well-structured HTMLs don't need anything fancy — regex handles it:

export default {
  async fetch(request) {
    const accept = request.headers.get("Accept") || "";
    if (!accept.includes("text/markdown")) {
      return fetch(request); // pass HTML straight to humans
    }

    // Agent asked for markdown. Fetch HTML from origin and convert.
    const response = await fetch(new Request(request.url, {
      headers: { Accept: "text/html" }
    }));
    const html = await response.text();

    const articleHtml = html.match(/<article[^>]*>([\s\S]*?)<\/article>/i)?.[1];
    const markdown = htmlToMarkdown(articleHtml);

    return new Response(frontmatter + markdown, {
      headers: {
        "Content-Type": "text/markdown; charset=utf-8",
        "x-markdown-tokens": String(Math.ceil(markdown.length / 4)),
        "Vary": "Accept",
      }
    });
  }
};

Result: an article HTML that costs ~53,000 tokens for an agent to parse becomes ~13,000 tokens in Markdown — 75% reduction. No Turndown, no JSDOM (neither runs in the Worker runtime), just regex and some care with <pre> and <code>.

Try it yourself:

# Humans get HTML
curl -sI https://adhenawer.net/posts/original/state-ai-2026.html | grep -i content-type
# content-type: text/html; charset=utf-8

# Agents get Markdown
curl -s https://adhenawer.net/posts/original/state-ai-2026.html \
  -H "Accept: text/markdown" | head -10
# ---
# title: "The State of AI in 2026"
# author: "Simon Willison · Lenny's Podcast"
# lang: en
# ---
# ## THE NOVEMBER INFLECTION POINT

Total cost to me: zero dollars. I wrote a technical post about this on the blog if you want details: Markdown for Agents on Free-Tier Cloudflare Workers.

References extracted automatically

Each article has a fixed sidebar with references that appear in the transcript. They're organized into categories:

  • Books → Amazon search
  • Tools → official site
  • Papers → arXiv or author's blog
  • People → Twitter/X (priority), fallback to Wikipedia
  • Concepts → Wikipedia
  • Companies → official site
  • Related posts → internal cross-link to the same theme in other articles

Extraction is done by Claude subagents in parallel — one per article, each receives the transcript and returns a structured JSON:

All together there are 714 extracted references across 21 posts. Of the 101 people referenced, 53 are linked to Twitter profiles (the rest are deceased academics or historical figures who obviously don't have Twitter).

Why bilingual?

Two reasons:

  1. Reach: most of the content I consume is in English, but I want Brazilian friends to be able to read it without the language barrier.
  2. SEO: properly configured hreflang alternates in sitemap + each HTML produce natural ranking in both languages.

Canonical URLs differ on each side:

  • PT: /posts/pt_br/chefe-do-claude-code-o-que-acontece-depois-que-a-programacao-for-resolvida.html
  • EN: /posts/original/head-claude-code-happens-after-coding-solved.html

Each slug is derived from the title in the corresponding language. The language switcher in the upper right of the header uses <link rel="alternate" hreflang> as the source of truth — zero hardcoded "URL A → URL B" mappings.

The daily flow inside Claude Code

The project runs almost entirely inside Claude Code (Anthropic's official CLI). I open the terminal, go into the repo, run claude, and say:

> add article from https://youtu.be/VIDEO_ID

The agent does the rest on its own: fetches the transcript, organizes into sections, translates to PT-BR, generates the HTML, updates docs/index.html, updates sitemap.xml, extracts references, commits, and pushes. I review the diff. If something looks off, I ask it to fix.

This works well for two reasons:

  1. CLAUDE.md — a markdown file at the root that contains the project's knowledge base (conventions, folder structure, slug format, etc.). That's what makes Claude Code "understand" the project without me re-explaining it each session.
  2. Subagents for parallel work — for large backfills (like generating English versions of the 20 articles that only existed in PT), the main agent spawns 20 subagents in parallel, each with isolated context, and aggregates the results. A backfill of 19 articles with ~270 thousand words of transcripts ran in 2 minutes that way. Main context stays clean.

Full stack

Total monthly cost: $0 (just the domain, which is annual).

  • Interface: Claude Code (CLI)
  • Model: Claude Opus 4.6
  • YouTube transcription: youtube-transcript-api
  • Twitter/X transcription: yt-dlp + mlx-whisper
  • Organization/translation: Claude
  • HTML build: pure Python (~400 lines)
  • Frontend: static HTML, CSS in one file
  • Hosting: GitHub Pages
  • Domain + CDN: Cloudflare (Free)
  • Edge / agents: Cloudflare Worker (Free) — Markdown on demand
  • Analytics: Umami Cloud (Free tier)

What's open

All the code is at github.com/adhenawer/video-to-text, MIT license. The files that matter:

  • src/pipeline.py — end-to-end orchestrator
  • src/providers/ — multi-provider abstraction (YouTube, Twitter/X). Easy to add Vimeo, TikTok, etc.
  • src/build_html.py — HTML generation
  • workers/markdown-agent/src/index.js — Cloudflare Worker
  • transcripts/ — original + translated transcripts + references JSONs

If anyone wants to run it for themselves: fork, clone, pip install -r requirements.txt, Claude Code, and send a URL in the chat.
