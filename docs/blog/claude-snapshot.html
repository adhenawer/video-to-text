claude-snapshot: levar sua config do Claude Code de uma máquina pra outra em 2 minutos

Rodolfo Moraes · 2026-04-19

O problema

Se você usa Claude Code, sabe que ~/.claude/ vira um organismo vivo. Plugins, hooks, skills, settings.json, CLAUDE.md global, statusline custom, marketplaces registrados. Tudo isso é seu. E tudo isso, historicamente, morria no momento em que você precisava trocar de máquina, formatar o Mac, ou entrar num computador de trabalho.

O claude-snapshot é um plugin do próprio Claude Code que exporta seu setup inteiro em um .tar.gz portátil e aplica em outra máquina com um comando.

Pra que serve

Quatro casos que eu tinha de verdade:

  • Duas máquinas. Pessoal em casa, de trabalho no escritório. Exportar num, dropar no Drive, aplicar no outro.
  • Backup antes de formatar. Tirar snapshot, wipe, reinstalar Claude Code, aplicar. Setup de volta.
  • Rollback de experimento. Vou testar um plugin novo que pode quebrar tudo? Snapshot antes. Deu ruim, volto pro estado conhecido-bom.
  • Onboarding. Entrou gente nova no time? Compartilha o snapshot, já vem com os mesmos plugins, hooks e convenções.

Como funciona

Quatro comandos dentro do próprio Claude Code:

/snapshot:export         # exporta pra ~/claude-snapshot-YYYY-MM-DD.tar.gz
/snapshot:inspect <path> # lê o manifesto sem extrair
/snapshot:diff <path>    # mostra o que vai mudar antes de aplicar
/snapshot:apply <path>   # aplica (com confirmação + .bak de tudo que sobrescreve)

Export normaliza caminhos absolutos (/Users/seunome/) pra $HOME, então o mesmo tarball funciona entre contas diferentes. Apply resolve de volta pro $HOME da máquina-destino e instala plugins faltantes via /plugin install do próprio Claude Code.

Os cinco princípios

A decisão de cada feature passa por isso:

P1 — Blast radius mínimo. One-shot, sem daemon, sem watch, sem rede. Tudo é comando explícito, escopo fechado em ~/.claude/ mais o tarball.

P2 — Allowlist, não denylist. Snapshot contém uma lista explícita do que é capturado. Arquivo novo que o Claude Code criar no futuro não entra automaticamente — atualização do plugin é que decide. Paga em manutenção manual, ganha em previsibilidade e auditabilidade. Tradeoff oposto ao "captura tudo que achar".

P3 — Sem rede. Nunca. Zero requisição de saída. Instalação de plugin no apply é delegada pro mecanismo oficial do Claude Code, que cuida da rede ele mesmo.

P4 — Diff antes de destruir. Todo apply mostra um resumo do que vai mudar, e você precisa confirmar. Todo arquivo sobrescrito vai pra <arquivo>.bak antes.

P5 — Cross-platform de verdade. Node.js, não bash. Mesmo código roda em macOS e Linux sem polyfill nem if $OSTYPE. Windows via WSL funciona; Windows nativo é best-effort.

O que migra e o que não

MigraNão migra
settings.json (plugins, hooks, permissions, env, statusLine)Sessões e histórico
CLAUDE.md global e outros .md de rootTelemetria
Manifests de plugin + marketplacesTokens OAuth de MCP
Scripts de hookPlugins escopados por projeto
MCP servers (só a config, sem OAuth token)
Cache de plugin (com --full)

MCP merece uma nota: o ~/.claude.json guarda tanto a config dos servers quanto tokens OAuth e estado de projetos. O plugin captura só a parte mcpServers e, no apply, reporta o que precisa ser instalado na máquina-destino em vez de sobrescrever o arquivo inteiro. Privacidade primeiro, automação segunda.

Anatomia do snapshot

claude-snapshot-YYYY-MM-DD.tar.gz
├── manifest.json           # schemaVersion, plugins, marketplaces, hooks, MDs, MCPs, checksums
├── settings.json           # suas settings do Claude Code
├── mcp-servers.json        # configs de MCP server (só se você tem algum; caminhos normalizados)
├── global-md/              # CLAUDE.md e outros .md no root
├── hooks/                  # seus scripts de hook
├── plugins/
│   ├── installed_plugins.json
│   ├── known_marketplaces.json
│   └── blocklist.json
└── cache/                  # (só com --full)

Instalação

/plugin marketplace add adhenawer/claude-snapshot
/plugin install snapshot@claude-snapshot
/reload-plugins

Por que escrevi

Ferramentas de dotfile-sync genéricas resolvem parte do problema. stow, chezmoi, yadm — todos funcionam. Mas ~/.claude/ tem particularidades: o installed_plugins.json referencia marketplaces que precisam estar registrados antes; o settings.json tem caminhos absolutos que quebram entre contas; os plugins têm sua própria lógica de install que não dá pra simular só copiando arquivo.

Então fiz específico, Node.js, MIT, sem telemetria, sem rede. Tarball auto-contido, inspecionável, com backup de tudo que toca.
