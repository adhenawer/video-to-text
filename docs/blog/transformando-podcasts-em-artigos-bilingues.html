Transformando podcasts de horas em artigos que dá pra ler

Rodolfo Moraes · 2026-04-14

Não sei se você também tem essa vibe, mas quando vejo um podcast novo de 2, 3, 4 horas no YouTube eu já sei: não vou ver. Não no celular encaixado no metrô, não no sofá depois do expediente, não na pausa do almoço. O problema não é falta de interesse — é que vídeo longo é um formato hostil pra quase tudo que não seja assistir do começo ao fim, sentado.

Vídeo é ruim pra:

  • Pesquisar: Ctrl+F não existe. Se você lembra vagamente que "aquele cara falou sobre vibe coding", boa sorte achando o trecho.
  • Citar: mandar um link com timestamp resolve mal, e transcrever a mão é trabalhoso.
  • Reler: não tem marcador visual, não tem progresso, não tem seções nomeadas.
  • Skim: você não consegue bater o olho e decidir se vale investir 3 horas.

Artigo resolve tudo isso. Mas quase nenhum podcast tem versão escrita decente. E legendas automáticas do YouTube são ilegíveis — um paredão de texto sem pontuação, cheio de "é, tipo, né, então".

O que construí

adhenawer.net é um site estático que pega vídeos do YouTube ou Twitter/X e transforma em artigos bilíngues organizados por seções temáticas. Português brasileiro e inglês, mesmo conteúdo, URLs canônicas em cada idioma.

No momento são 20 artigos publicados, cobrindo gente como Boris Cherny (Claude Code), Simon Willison, Steve Yegge, Andrew Ng na Stanford, Dario Amodei, Robert Shiller (Yale, sobre finanças comportamentais), Patrick Winston (MIT, aquela palestra clássica de How to Speak), e mais.

Cada artigo tem:

  • Título e subtítulo limpos
  • Índice clicável
  • Seções temáticas (não cronológicas — reorganizadas por assunto)
  • Barra de progresso de leitura
  • "Continuar de onde parou" salvo por dispositivo
  • Sidebar de referências: livros citados linkados no Amazon, ferramentas com site oficial, pessoas com Twitter, conceitos com Wikipedia, posts relacionados
  • Switcher de idioma PT/EN no header

Zero framework. HTML estático servido no GitHub Pages. Carrega rápido em qualquer coisa, inclusive 3G sofrido.

A pipeline

URL do vídeo (YouTube ou Twitter/X)
  ↓
Detecta provider pela URL
  ↓
YouTube: youtube-transcript-api (legendas oficiais quando existem)
Twitter/X: yt-dlp baixa áudio → mlx-whisper transcreve local
  ↓
Transcrição bruta em inglês (ou idioma original do vídeo)
  ↓
Claude (organização): lê transcrição, remove "né", "tipo", timestamps,
                      sponsor reads, reorganiza em seções temáticas,
                      produz texto natural em PT-BR e em EN
  ↓
Python gera HTML com JSON-LD, Open Graph, hreflang, sidebar de referências
  ↓
Git commit + push → GitHub Pages publica

Sem CMS, sem banco de dados, sem etapa de build em frontend. O script src/build_html.py tem ~400 linhas de Python puro e escreve HTML semântico direto:

return f'''<!DOCTYPE html>
<html lang="{html_lang}">
<head>
<link rel="alternate" hreflang="pt-BR" href="{pt_href}">
<link rel="alternate" hreflang="en" href="{en_href}">
<script type="application/ld+json">{jsonld}</script>
<link rel="stylesheet" href="../../css/style.css">
</head>
<body class="page-article">
<header class="site-header">...</header>
<article class="container">
  <h1>{title}</h1>
  <nav><h3>Conteúdo</h3><ol>{toc_html}</ol></nav>
  {body_html}
</article>
{references_sidebar}
</body>
</html>'''

O Whisper local

Pra vídeos do Twitter/X (ou YouTube quando a legenda oficial está bloqueada) o pipeline usa mlx-whisper — a implementação MLX do Whisper, otimizada nativa pro Apple Silicon. Custo de transcrição: zero. Velocidade: ~5× real-time num M1 Pro. Um podcast de 1 hora transcreve em uns 12 minutos.

import mlx_whisper

result = mlx_whisper.transcribe(
    audio_path,
    path_or_hf_repo="mlx-community/whisper-large-v3-turbo",
    word_timestamps=False,
)

Se você está em Linux ou Windows dá pra trocar por faster-whisper (CTranslate2), mesma ideia. A única coisa que importa é não mandar áudio pra API de ninguém pra economizar grana — Whisper rodando local é bom o suficiente e é grátis.

A parte que mais me divirto: Markdown para agentes

Essa é a mais massa. A Cloudflare tem uma feature chamada Markdown for Agents que converte HTML em Markdown na edge quando um agente de IA faz a requisição com Accept: text/markdown. O problema é que essa feature só existe no plano pago (Pro ou Business).

Eu dei uma olhada no Worker runtime deles e percebi que dá pra replicar em ~150 linhas de JavaScript rodando no plano Free (100k requests/dia). A chave é que HTMLs bem estruturados não precisam de nada muito sofisticado — regex resolve:

export default {
  async fetch(request) {
    const accept = request.headers.get("Accept") || "";
    if (!accept.includes("text/markdown")) {
      return fetch(request); // passa HTML direto pra humanos
    }

    // Agente pediu markdown. Busca HTML do origin e converte.
    const response = await fetch(new Request(request.url, {
      headers: { Accept: "text/html" }
    }));
    const html = await response.text();

    const articleHtml = html.match(/<article[^>]*>([\s\S]*?)<\/article>/i)?.[1];
    const markdown = htmlToMarkdown(articleHtml);

    return new Response(frontmatter + markdown, {
      headers: {
        "Content-Type": "text/markdown; charset=utf-8",
        "x-markdown-tokens": String(Math.ceil(markdown.length / 4)),
        "Vary": "Accept",
      }
    });
  }
};

Resultado: um HTML de artigo que custa ~53.000 tokens pra um agente parsear vira ~13.000 tokens em Markdown — redução de 75%. Sem Turndown, sem JSDOM (nenhum dos dois roda em Worker runtime), só regex e uns cuidados com <pre> e <code>.

Testa você mesmo:

# Humano recebe HTML
curl -sI https://adhenawer.net/posts/original/state-ai-2026.html | grep -i content-type
# content-type: text/html; charset=utf-8

# Agente recebe Markdown
curl -s https://adhenawer.net/posts/original/state-ai-2026.html \
  -H "Accept: text/markdown" | head -10
# ---
# title: "The State of AI in 2026"
# author: "Simon Willison · Lenny's Podcast"
# lang: en
# ---
# ## THE NOVEMBER INFLECTION POINT

Custo total pra mim: zero dólares. Escrevi um post técnico sobre isso aqui no blog se quiser detalhes: Markdown para agentes no Cloudflare grátis.

Referências extraídas automaticamente

Cada artigo tem uma sidebar fixa com referências que aparecem na transcrição. São organizadas em categorias:

  • Livros → busca no Amazon
  • Ferramentas → site oficial
  • Papers → arXiv ou blog do autor
  • Pessoas → Twitter/X (prioridade), fallback pra Wikipedia
  • Conceitos → Wikipedia
  • Empresas → site oficial
  • Posts relacionados → cross-link interno pro mesmo tema em outros artigos

A extração é feita por subagentes do Claude em paralelo — um por artigo, cada um recebe a transcrição e devolve um JSON estruturado:

Por que bilíngue?

Dois motivos:

  1. Alcance: boa parte do conteúdo que consumo é em inglês, mas quero que amigos BR leiam sem barreira de idioma.
  2. SEO: hreflang alternates corretamente configurados em sitemap + cada HTML geram ranqueamento natural em ambos idiomas.

URLs canônicas são diferentes em cada lado:

  • PT: /posts/pt_br/chefe-do-claude-code-o-que-acontece-depois-que-a-programacao-for-resolvida.html
  • EN: /posts/original/head-claude-code-happens-after-coding-solved.html

Cada slug é derivado do título no idioma correspondente. Switcher de idioma na lateral direita do header usa o <link rel="alternate" hreflang> como fonte da verdade — zero configuração hardcoded de "URL A → URL B".

O fluxo diário no Claude Code

O projeto roda quase inteiramente dentro do Claude Code (a CLI oficial da Anthropic). Eu abro o terminal, entro no repo, chamo claude e digo:

> adiciona artigo de https://youtu.be/VIDEO_ID

O agente faz o resto sozinho: busca transcrição, organiza em seções, traduz pra PT-BR, gera o HTML, atualiza o docs/index.html, atualiza o sitemap.xml, extrai referências, commita e faz push. Eu revejo o diff. Se algo ficou estranho, peço pra corrigir.

Isso funciona bem por dois motivos:

  1. CLAUDE.md — um arquivo de markdown na raiz que contém a base de conhecimento do projeto (convenções, estrutura de pastas, formato de slug, etc.). É o que faz o Claude Code "entender" o projeto sem eu precisar re-explicar a cada sessão.
  2. Subagentes pra trabalhos paralelos — pra backfills grandes (tipo gerar versão inglesa dos 20 artigos que só existiam em PT), o agente principal dispara 20 subagentes em paralelo, cada um com contexto isolado, e agrega os resultados. Um backfill de 19 artigos com ~270 mil palavras de transcrições rodou em 2 minutos desse jeito. Contexto do main fica limpo.

Stack completo

Custo mensal total: $0 (só o domínio, que é anual).

  • Interface: Claude Code (CLI)
  • Modelo: Claude Opus 4.6
  • Transcrição YouTube: youtube-transcript-api
  • Transcrição Twitter/X: yt-dlp + mlx-whisper
  • Organização/tradução: Claude
  • Build HTML: Python puro (~400 linhas)
  • Frontend: HTML estático, CSS em um arquivo
  • Hosting: GitHub Pages
  • Domínio + CDN: Cloudflare (Free)
  • Edge / agentes: Cloudflare Worker (Free) — Markdown on demand
  • Analytics: Umami Cloud (Free tier)

O que tem de aberto

Código todo em github.com/adhenawer/video-to-text, licença MIT. Os arquivos que importam:

  • src/pipeline.py — orquestrador de ponta a ponta
  • src/providers/ — abstração multi-provider (YouTube, Twitter/X). Fácil adicionar Vimeo, TikTok, etc.
  • src/build_html.py — geração de HTML
  • workers/markdown-agent/src/index.js — Worker do Cloudflare
  • transcripts/ — transcrições originais + traduzidas + JSONs de referências

Se alguém quiser rodar pra si, fork, clone, pip install -r requirements.txt, Claude Code, e manda uma URL no chat.

Convite

Não estou fazendo grana com isso. É um sistema pra eu mesmo consumir conteúdo bom no formato que funciona pra mim. Se serve pra mais alguém, melhor.

Se você quiser contribuir com um novo provider (tipo Vimeo ou podcast feeds), bugs no CSS, ou sugerir um vídeo pra transcrever, abre uma issue ou PR. Se só quiser ler, o índice está em adhenawer.net (PT) e adhenawer.net/en/ (EN).

Quem me deu o pontapé inicial foi uma conversa sobre como o Claude Code mudou o que "um engenheiro consegue fazer num fim de semana". Esse projeto é, em boa parte, uma demonstração disso. Também é uma desculpa pra consumir mais podcast bom sem precisar passar 4 horas de fone.

Se você tem uma dor parecida — conteúdo bom em formato ruim — talvez valha o detour.
