Markdown para Agentes no plano gratuito do Cloudflare Workers

video-to-text · 2026-04-13

A Cloudflare oferece um recurso chamado Markdown for Agents que devolve uma versão limpa em Markdown de qualquer página quando o cliente envia Accept: text/markdown. Ele está restrito aos planos Pro e Business. Para um blog estático hospedado no GitHub Pages, pagar por um plano de edge só para ajudar crawlers a interpretarem artigos é difícil de justificar. Este post percorre um Cloudflare Worker de 170 linhas que replica o recurso no plano gratuito para adhenawer.net e explica os trade-offs envolvidos.

Por que servir Markdown

Cada artigo deste site é um arquivo HTML autocontido com seletor de tema, barra de progresso, banner de retomada, JSON-LD, tags Open Graph e navegação inline. Um único artigo pesa cerca de 53.000 tokens quando um LLM ingere o HTML cru. A maior parte disso é marcação que o modelo vai descartar. Servir o mesmo conteúdo em Markdown com um frontmatter YAML reduz o mesmo artigo para aproximadamente 13.000 tokens. Isso é uma redução de 75%, o que importa para qualquer pipeline de retrieval que paga por token ou tem uma janela de contexto finita.

O truque é content negotiation. Um navegador pede text/html e recebe o HTML completo. Um agente pede text/markdown e recebe Markdown. A URL continua a mesma. Sem endpoints separados, sem duplicação de sitemap, sem novos caminhos para indexar.

O Worker em uma leitura

O entry point verifica o header Accept. Se ele não inclui text/markdown, o Worker simplesmente faz proxy da requisição para a origem e devolve o que o GitHub Pages retornou. Sem transformação, sem overhead além de um fetch.

function wantsMarkdown(request) {
  const accept = request.headers.get("Accept") || "";
  return accept.includes("text/markdown");
}

export default {
  async fetch(request, env, ctx) {
    if (!wantsMarkdown(request)) {
      return fetch(request);
    }
    // ...fetch origin as HTML, transform, return Markdown
  }
};

Quando Markdown é solicitado, o Worker busca a mesma URL na origem forçando Accept: text/html, extrai o bloco <article>, remove o chrome e converte o que sobra.

Por que regex em vez de um parser

O caminho óbvio para converter HTML em Markdown é usar Turndown com JSDOM. Nenhum dos dois roda bem no runtime dos Workers. O JSDOM puxa built-ins do Node que não estão disponíveis, e empacotar parsers de DOM estoura o limite de 1 MB de script do Worker no plano gratuito. A Cloudflare oferece o HTMLRewriter, que é rápido e streaming, mas ele foi desenhado para reescritas pontuais e não para converter a árvore inteira em texto.

Como o HTML de origem é gerado pelo meu próprio build_html.py, a estrutura é previsível. Uma sequência de passes de regex é suficiente. Isso não é um conversor de uso geral. É um conversor para exatamente o HTML que este site produz.

// Remove elements we don't want
md = md.replace(/<div class="theme-bar">[\s\S]*?<\/div>/gi, "");
md = md.replace(/<a[^>]*class="back-home"[^>]*>[\s\S]*?<\/a>/gi, "");
md = md.replace(/<script[\s\S]*?<\/script>/gi, "");
md = md.replace(/<style[\s\S]*?<\/style>/gi, "");
md = md.replace(/<nav[\s\S]*?<\/nav>/gi, "");

// Convert headings
md = md.replace(/<h2[^>]*>([\s\S]*?)<\/h2>/gi, "## $1\n\n");

// Convert links
md = md.replace(/<a[^>]*href="([^"]*)"[^>]*>([\s\S]*?)<\/a>/gi, "[$2]($1)");

// Strip whatever tags remain
md = md.replace(/<[^>]+>/g, "");

A ordem importa. Links precisam ser convertidos antes do removedor genérico de tags, senão o href é perdido. Headings precisam ser convertidos antes dos parágrafos, senão o texto interno é embrulhado duas vezes. Blocos de script e style precisam ser deletados primeiro porque contêm caracteres que confundiriam os regex de âncoras e atributos.

Frontmatter e o header de tokens

O Worker extrai <title>, meta name="description" e meta name="author" do HTML original e monta um bloco de frontmatter YAML. Ele também emite um header de resposta x-markdown-tokens com uma estimativa de contagem de tokens (comprimento dividido por quatro, suficiente para orçamento).

return new Response(finalMarkdown, {
  status: 200,
  headers: {
    "Content-Type": "text/markdown; charset=utf-8",
    "x-markdown-tokens": String(tokens),
    "Vary": "Accept",
    "Cache-Control": "public, max-age=3600",
    "Access-Control-Allow-Origin": "*",
  },
});

O header Vary: Accept não é opcional. Sem ele, qualquer cache na frente do Worker, incluindo o da própria Cloudflare, pode servir a versão HTML para um agente ou a versão Markdown para um navegador, dependendo de quem pediu primeiro. Com Vary: Accept, o cache chaveia pelo header e as duas variantes coexistem.

Custo e limites

O plano gratuito dos Workers oferece 100.000 requisições por dia e 10 ms de CPU por requisição. O pipeline de regex em um arquivo HTML de 50 KB roda em poucos milissegundos. Um blog pequeno nunca chega perto de nenhum dos tetos. Egress é gratuito e não há cobrança pela subrequest ao GitHub Pages.

Os trade-offs merecem ser nomeados. O conversor é frágil contra HTML para o qual não foi desenhado. Adicione um novo componente ao template sem atualizar o Worker e o Markdown resultante vai conter marcação vazada ou conteúdo ausente. Não há suíte de testes no Worker; o ciclo de feedback é um curl no terminal. E regex não lida bem com listas aninhadas ou tabelas. Para um blog de transcrições onde os artigos são parágrafos, headings e figuras ocasionais, tudo bem. Para um site de documentação com exemplos de código e tabelas, um parser de verdade valeria o setup extra.

Experimente

Mesma URL, Accept diferente, uma fração dos tokens. Nenhum upgrade de plano necessário.
